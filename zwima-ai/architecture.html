Request-Flow (V1)

  1. Kunde sendet Anfrage mit Zwima API Key an das API Gateway
  2. Authentication Layer validiert Schlüssel, Konto und Rate Limits
  3. Credit Engine prüft Credit Balance und Monthly Quota (Pre-Auth)
  4. Model Router wählt Provider via Smart Rules, Cost Optimization oder Fallback
  5. Provider Adapter transformiert Request und ruft externe API auf
  6. Response wird normalisiert; Credit Engine führt Deduction und Overage Calculation durch
  7. Audit & Logging protokolliert Metadaten asynchron (ohne Prompt-Inhalte)

Customer Account

Zentrale Entität für jeden Geschäftskunden. Verknüpft Subscription Plan, API-Schlüssel, Credit Balance und Billing Records.

CustomerAccount {
  id: UUID
  company_name: string
  email: string
  status: active | suspended | closed
  subscription_plan_id: FK
  credit_balance: int
  created_at: timestamp
}

Subscription Plan

Definiert monatliches API-Nutzungspaket, Credit-Kontingent (Monthly Quota), verfügbare Modelle und Overage-Regeln.

Zwima API Key

Client-seitige Authentifizierung. Validierung erfolgt in der Authentication Layer, nicht im Gateway selbst.

zwima_sk_live_<32 Zeichen>
zwima_sk_test_<32 Zeichen>

Schlüssel werden gehasht gespeichert. Widerruf invalidiert den Cache innerhalb von 60 Sekunden.

API Gateway

Einziger öffentlicher Endpunkt. Delegiert Authentifizierung, Credit-Prüfung und Routing an interne Module.

POST /v1/chat/completions
Authorization: Bearer zwima_sk_live_...
Content-Type: application/json

Das Gateway führt TLS-Terminierung, Request-Schema-Validierung und Response-Normalisierung durch.

Authentication Layer

Erste interne Schicht nach dem API Gateway. Jede Anfrage muss alle drei Prüfungen bestehen, bevor sie weitergeleitet wird.

API Key Validation

  • Bearer-Token extrahieren und Format prüfen
  • Hash-Vergleich gegen Datenbank
  • Widerrufsstatus aus Redis-Cache lesen
  • API Key → Customer Account auflösen

User Account Validation

  • Kontostatus active erforderlich
  • Subscription Plan gültig und nicht abgelaufen
  • Modell-Berechtigung gemäß Plan prüfen
  • Test-Schlüssel nur auf Test-Modelle

Rate Limiting

  • Token-Bucket pro API Key (Standard: 60 req/min)
  • Konto-weites Tageslimit gemäß Plan
  • HTTP 429 bei Überschreitung mit Retry-After
  • Enterprise: konfigurierbare Limits

Credit Engine

Zentrale Verbrauchs- und Kontingentverwaltung. Ersetzt die isolierten Module „Usage Tracking" und „Credits Deduction" als einheitliche Engine.

Credit Balance

Echtzeit-Saldo des Kunden. Wird nach jeder Deduction atomar aktualisiert (optimistic locking).

Monthly Quota

Monatliches Kontingent aus Subscription Plan. Reset am ersten Tag des Abrechnungszeitraums.

Credit Deduction

Post-Request: credits = f(input_tokens, output_tokens, model_factor). Asynchron via Event Queue.

Overage Calculation

Bei Business+/Enterprise: Verbrauch über Quota wird erfasst und für Overage Invoice vorbereitet.

CreditTransaction {
  id: UUID
  account_id: FK
  type: deduction | topup | refund | overage
  amount: int
  balance_after: int
  usage_event_id: FK?
  created_at: timestamp
}

Model Router

Intelligente Weiterleitung von Anfragen an den optimalen Provider. Ersetzt den früheren „Provider Router" als eigenständiges Modul mit erweiterter Logik.

Smart Routing Rules

  • Explizites Modell im Request → direktes Mapping
  • Alias-Auflösung (gpt-4 → gpt-4o)
  • Plan-basierte Modell-Whitelist

Cost Optimization

  • Bei model: auto: günstigstes Modell für Task-Typ
  • Interne Kosten-Matrix (nicht öffentlich)
  • Enterprise: kundenspezifische Optimierungsprofile

Fallback Routing

  • Provider-Health-Check alle 30s
  • Automatischer Failover auf konfiguriertes Backup-Modell
  • Max. 2 Retries mit exponentiellem Backoff

Provider Adapter Layer

Ein Adapter pro externem Provider. Übersetzt das einheitliche Zwima-Request-Format in provider-spezifische API-Aufrufe und normalisiert Responses.

OpenAI Adapter

Chat Completions, Embeddings. Mapping: Zwima → OpenAI /v1/chat/completions

Claude Adapter

Messages API. Mapping: Zwima Messages → Anthropic /v1/messages

DeepSeek Adapter

OpenAI-kompatibles Format. Endpoint-Routing nach Region.

Gemini Adapter

Google Generative AI. Multimodal-Parameter-Transformation.

Qwen Adapter

DashScope API. Mehrsprachige Token-Zählung.

Adapter-Interface (V1)

interface ProviderAdapter {
  provider_id: string
  send(request: ZwimaRequest): Promise<ZwimaResponse>
  healthCheck(): Promise<boolean>
  estimateTokens(request: ZwimaRequest): int
}

Audit & Logging

Querschnittsmodul. Empfängt Events aus allen Schichten asynchron. Keine Prompt-Inhalte in Logs (GDPR).

API Requests

  • Request-ID, Timestamp, Account, API Key, Modell
  • Input/Output Token-Anzahl, Latenz, HTTP-Status
  • Retention: 13 Monate

Billing Events

  • Credit Deductions, Overage, Invoice-Erstellung
  • Immutable Audit Trail für Abrechnungsstreitigkeiten
  • Retention: 7 Jahre (gesetzlich)

Security Logs

  • Fehlgeschlagene Authentifizierung, Rate-Limit-Hits
  • API-Key-Widerrufe, Admin-Aktionen
  • Retention: 90 Tage

Admin Console

Interne Verwaltungsoberfläche für Zwima-Mitarbeiter. Nicht öffentlich zugänglich. Authentifizierung via SSO + RBAC.

Customer Management

  • Konten anlegen, sperren, schließen
  • Plan-Zuweisung und Upgrade/Downgrade
  • API-Key-Übersicht (nur Prefix sichtbar)

Credit Management

  • Manuelle Credit-Topups und Gutschriften
  • Quota-Anpassungen für Enterprise
  • Verbrauchshistorie pro Kunde

Billing Management

  • Rechnungen erstellen und freigeben
  • Overage-Review vor Rechnungsstellung
  • Zahlungsstatus und Mahnwesen

Provider Monitoring

  • Live-Status aller Provider und Adapter
  • Latenz- und Fehlerrate-Dashboards
  • Failover-Events und Incident-Log

Usage & Billing

Usage Events werden von der Credit Engine erzeugt und von Audit & Logging persistiert. Billing Records entstehen am Periodenende.

UsageEvent → CreditTransaction → BillingRecord

BillingRecord {
  id: UUID
  account_id: FK
  type: subscription | overage | credit_note
  amount_eur: decimal
  period_start / period_end: date
  status: draft | issued | paid
}

GDPR Data Handling

Architektonische Grundlage unter allen Schichten. Gilt für Gateway, interne Module, Audit & Logging und Admin Console.

PrinzipUmsetzung V1
DatenminimierungNur Metadaten in Logs; keine Prompt-Speicherung
Transit-onlyPrompts In-Memory; max. 24h anonymisiert bei Fehleranalyse
EU-InfrastrukturPrimäre Verarbeitung in EU-Rechenzentren
AVV (DPA)Auf Anfrage für alle Geschäftskunden
LöschungKontolöschung innerhalb 30 Tage
