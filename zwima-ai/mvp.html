Feature-Details

1. Customer Registration

  • Registrierungsformular: Firmenname, E-Mail, Passwort, Ansprechpartner
  • E-Mail-Verifizierung vor erstem API-Zugang
  • Automatische Anlage von Customer Account mit Status pending → active
  • Standard-Plan: Starter (49 EUR/Monat, 10.000 Credits)

Kein Self-Service-Planwechsel in V0.1 – nur via Admin.

2. Login

  • E-Mail + Passwort, Session-basiert (HTTP-only Cookie)
  • Passwort-Reset per E-Mail-Link
  • Zugang zum Kunden-Dashboard nach Login

Kein SSO, kein 2FA in V0.1.

3. API Key Generation

  • Kunde erstellt Schlüssel im Dashboard (Name + Live/Test)
  • Format: zwima_sk_live_ / zwima_sk_test_
  • Einmalige Anzeige bei Erstellung; danach nur Prefix sichtbar
  • Widerruf mit sofortiger Wirkung
  • Max. 3 aktive Schlüssel pro Konto (V0.1)

4. OpenAI Integration Only

  • Ein Endpunkt: POST /v1/chat/completions
  • Unterstützte Modelle: gpt-4o, gpt-4o-mini, gpt-3.5-turbo
  • Direkter OpenAI Adapter – kein Model Router
  • Request/Response 1:1 an OpenAI-Format angelehnt
  • Fehler von OpenAI werden durchgereicht (normalisierte Fehlercodes)
POST /v1/chat/completions
Authorization: Bearer zwima_sk_live_...
{ "model": "gpt-4o-mini", "messages": [...] }

5. Credit Balance

  • Monatliches Kontingent gemäß Plan (Starter: 10.000 Credits)
  • Anzeige im Dashboard: Gesamt, Verbraucht, Verbleibend
  • Pre-Request-Prüfung: HTTP 429 bei leerem Kontingent
  • Credit-Faktor fest für V0.1 (eine Rate pro Modell, intern konfiguriert)
credits_used = ceil((input_tokens + output_tokens) * model_factor)

6. Usage Tracking

  • Jeder API-Aufruf erzeugt ein Usage Event (asynchron)
  • Gespeichert: Timestamp, Account, API Key, Modell, Tokens, Status, Latenz
  • Nicht gespeichert: Prompt-Inhalte, Antworttexte
  • Dashboard: Tages- und Monatsübersicht (einfache Tabelle)

7. Billing Records

  • Monatliche Subscription-Rechnung (manuell oder halbautomatisch)
  • Rechnungsdatensatz: Nummer, Betrag, Zeitraum, Status (draft / issued / paid)
  • PDF-Export manuell in V0.1 (Template + Admin-Aktion)
  • Kein Zahlungsgateway in V0.1 – Rechnung per E-Mail, Zahlung per Überweisung

Overage-Berechnung nicht in V0.1 – harter Stopp bei Quota.

8. Admin Panel

  • Internes Tool (separate URL, Basic Auth oder einfaches Login)
  • Kundenliste: Suchen, Status ändern (active / suspended)
  • Credit-Topup manuell (z. B. bei Zahlungseingang)
  • Rechnung erstellen und als „issued" markieren
  • Usage-Übersicht pro Kunde (letzte 30 Tage)
  • OpenAI-Status-Anzeige (letzter Health-Check)

Kein Provider Monitoring Dashboard – nur OpenAI Ping.

Launch-Kriterien

MVP V0.1 gilt als launch-ready, wenn alle Punkte erfüllt sind:

  • Kunde kann sich registrieren, E-Mail verifizieren und einloggen
  • Kunde kann API-Schlüssel erstellen und widerrufen
  • API-Aufruf an gpt-4o-mini liefert valide OpenAI-Response
  • Credits werden korrekt abgezogen; Dashboard zeigt aktuellen Stand
  • Bei leerem Kontingent: HTTP 429, kein stiller Durchgriff
  • Usage Events werden persistiert (ohne Prompt-Inhalte)
  • Admin kann Kunde aktivieren, Credits aufladen, Rechnung ausstellen
  • Erster Testkunde hat bezahlt und produktiv API-Aufrufe durchgeführt
